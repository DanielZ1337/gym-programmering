\hypertarget{class_fl___graphics___driver}{}\doxysection{Fl\+\_\+\+Graphics\+\_\+\+Driver Class Reference}
\label{class_fl___graphics___driver}\index{Fl\_Graphics\_Driver@{Fl\_Graphics\_Driver}}


A virtual class subclassed for each graphics driver F\+L\+TK uses. Typically, F\+L\+TK applications do not use directly objects from this class. Rather, they perform drawing operations (e.\+g., \mbox{\hyperlink{class_fl___graphics___driver_a2986a868e9cc8d9141acde94c0fe8ab0}{fl\+\_\+rectf()}}) that operate on the current drawing surface (see \mbox{\hyperlink{class_fl___surface___device}{Fl\+\_\+\+Surface\+\_\+\+Device}}). Drawing operations are functionally presented in drawing and as function lists in the \mbox{\hyperlink{group__fl__drawings}{Drawing functions}} and \mbox{\hyperlink{group__fl__attributes}{Color \& Font functions}} modules. The \mbox{\hyperlink{_fl___device_8_h_a70e4f05429b26bb420c70f64e972f2a5}{fl\+\_\+graphics\+\_\+driver}} global variable gives at any time the graphics driver used by all drawing operations. Its value changes when drawing operations are directed to another drawing surface by \mbox{\hyperlink{class_fl___surface___device_a6f88aed1b68854433d60cf23731c3d79}{Fl\+\_\+\+Surface\+\_\+\+Device\+::set\+\_\+current()}}.

\doxysubsection{Detailed Description}
A virtual class subclassed for each graphics driver F\+L\+TK uses. Typically, F\+L\+TK applications do not use directly objects from this class. Rather, they perform drawing operations (e.\+g., \mbox{\hyperlink{class_fl___graphics___driver_a2986a868e9cc8d9141acde94c0fe8ab0}{fl\+\_\+rectf()}}) that operate on the current drawing surface (see \mbox{\hyperlink{class_fl___surface___device}{Fl\+\_\+\+Surface\+\_\+\+Device}}). Drawing operations are functionally presented in drawing and as function lists in the \mbox{\hyperlink{group__fl__drawings}{Drawing functions}} and \mbox{\hyperlink{group__fl__attributes}{Color \& Font functions}} modules. The \mbox{\hyperlink{_fl___device_8_h_a70e4f05429b26bb420c70f64e972f2a5}{fl\+\_\+graphics\+\_\+driver}} global variable gives at any time the graphics driver used by all drawing operations. Its value changes when drawing operations are directed to another drawing surface by \mbox{\hyperlink{class_fl___surface___device_a6f88aed1b68854433d60cf23731c3d79}{Fl\+\_\+\+Surface\+\_\+\+Device\+::set\+\_\+current()}}. 

{\ttfamily The} \mbox{\hyperlink{class_fl___graphics___driver}{Fl\+\_\+\+Graphics\+\_\+\+Driver}} class is of interest if one wants to perform new kinds of drawing operations. An example would be to draw to a P\+DF file. This would involve creating a new \mbox{\hyperlink{class_fl___graphics___driver}{Fl\+\_\+\+Graphics\+\_\+\+Driver}} derived class. Draw ellipse sections using integer coordinates.

These functions match the rather limited circle drawing code provided by X and W\+I\+N32. The advantage over using fl\+\_\+arc with floating point coordinates is that they are faster because they often use the hardware, and they draw much nicer small circles, since the small sizes are often hard-\/coded bitmaps.

If a complete circle is drawn it will fit inside the passed bounding box. The two angles are measured in degrees counter-\/clockwise from 3 o\textquotesingle{}clock and are the starting and ending angle of the arc, {\ttfamily a2} must be greater or equal to {\ttfamily a1}.

\mbox{\hyperlink{class_fl___graphics___driver_a55e75228acb8f4c0febbc72fc9ab50dd}{fl\+\_\+arc()}} draws a series of lines to approximate the arc. Notice that the integer version of \mbox{\hyperlink{class_fl___graphics___driver_a55e75228acb8f4c0febbc72fc9ab50dd}{fl\+\_\+arc()}} has a different number of arguments than the double version \mbox{\hyperlink{class_fl___graphics___driver_a55e75228acb8f4c0febbc72fc9ab50dd}{fl\+\_\+arc(double x, double y, double r, double start, double end)}}

The callback function {\ttfamily cb} is called with the {\ttfamily void$\ast$} {\ttfamily data} user data pointer to allow access to a structure of information about the image, and the {\ttfamily x}, {\ttfamily y}, and {\ttfamily w} of the scan line desired from the image. 0,0 is the upper-\/left corner of the image, not {\ttfamily x}, {\ttfamily y}. A pointer to a buffer to put the data into is passed. You must copy {\ttfamily w} pixels from scanline {\ttfamily y}, starting at pixel {\ttfamily x}, to this buffer.

Due to cropping, less than the whole image may be requested. So {\ttfamily x} may be greater than zero, the first {\ttfamily y} may be greater than zero, and {\ttfamily w} may be less than {\ttfamily W}.
