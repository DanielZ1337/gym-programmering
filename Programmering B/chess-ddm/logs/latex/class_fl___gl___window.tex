\doxysubsection{Detailed Description}
The \mbox{\hyperlink{class_fl___gl___window}{Fl\+\_\+\+Gl\+\_\+\+Window}} widget sets things up so Open\+GL works.

It also keeps an Open\+GL \char`\"{}context\char`\"{} for that window, so that changes to the lighting and projection may be reused between redraws. \mbox{\hyperlink{class_fl___gl___window}{Fl\+\_\+\+Gl\+\_\+\+Window}} also flushes the Open\+GL streams and swaps buffers after \mbox{\hyperlink{class_fl___gl___window_ab3c931bb0ae05687baf9928011ceaf94}{draw()}} returns.

Open\+GL hardware typically provides some overlay bit planes, which are very useful for drawing UI controls atop your 3D graphics. If the overlay hardware is not provided, F\+L\+TK tries to simulate the overlay. This works pretty well if your graphics are double buffered, but not very well for single-\/buffered.

Please note that the F\+L\+TK drawing and clipping functions will not work inside an \mbox{\hyperlink{class_fl___gl___window}{Fl\+\_\+\+Gl\+\_\+\+Window}}. All drawing should be done using Open\+GL calls exclusively. Even though \mbox{\hyperlink{class_fl___gl___window}{Fl\+\_\+\+Gl\+\_\+\+Window}} is derived from \mbox{\hyperlink{class_fl___group}{Fl\+\_\+\+Group}}, it is not useful to add other F\+L\+TK Widgets as children, unless those widgets are modified to draw using Open\+GL calls.

Set or change the Open\+GL capabilites of the window. The value can be any of the following OR\textquotesingle{}d together\+:


\begin{DoxyItemize}
\item {\ttfamily F\+L\+\_\+\+R\+GB} -\/ R\+GB color (not indexed)
\item {\ttfamily F\+L\+\_\+\+R\+G\+B8} -\/ R\+GB color with at least 8 bits of each color
\item {\ttfamily F\+L\+\_\+\+I\+N\+D\+EX} -\/ Indexed mode
\item {\ttfamily F\+L\+\_\+\+S\+I\+N\+G\+LE} -\/ not double buffered
\item {\ttfamily F\+L\+\_\+\+D\+O\+U\+B\+LE} -\/ double buffered
\item {\ttfamily F\+L\+\_\+\+A\+C\+C\+UM} -\/ accumulation buffer
\item {\ttfamily F\+L\+\_\+\+A\+L\+P\+HA} -\/ alpha channel in color
\item {\ttfamily F\+L\+\_\+\+D\+E\+P\+TH} -\/ depth buffer
\item {\ttfamily F\+L\+\_\+\+S\+T\+E\+N\+C\+IL} -\/ stencil buffer
\item {\ttfamily F\+L\+\_\+\+M\+U\+L\+T\+I\+S\+A\+M\+P\+LE} -\/ multisample antialiasing
\item {\ttfamily F\+L\+\_\+\+O\+P\+E\+N\+G\+L3} -\/ use Open\+GL version 3.\+0 or more.
\end{DoxyItemize}

F\+L\+\_\+\+R\+GB and F\+L\+\_\+\+S\+I\+N\+G\+LE have a value of zero, so they are \char`\"{}on\char`\"{} unless you give F\+L\+\_\+\+I\+N\+D\+EX or F\+L\+\_\+\+D\+O\+U\+B\+LE.

If the desired combination cannot be done, F\+L\+TK will try turning off F\+L\+\_\+\+M\+U\+L\+T\+I\+S\+A\+M\+P\+LE. If this also fails the \mbox{\hyperlink{class_fl___gl___window_a05f9c5395325f1f5a905207f0ccb994a}{show()}} will call \mbox{\hyperlink{group__group__comdlg_ga9acd48aa40a8fb587f361a1f1907f207}{Fl\+::error()}} and not show the window.

You can change the mode while the window is displayed. This is most useful for turning double-\/buffering on and off. Under X this will cause the old X window to be destroyed and a new one to be created. If this is a top-\/level window this will unfortunately also cause the window to blink, raise to the top, and be de-\/iconized, and the xid() will change, possibly breaking other code. It is best to make the GL window a child of another window if you wish to do this!

\mbox{\hyperlink{png_8h_a3eabae3e1e65723a7ec3445646cb7795}{mode()}} must not be called within \mbox{\hyperlink{class_fl___gl___window_ab3c931bb0ae05687baf9928011ceaf94}{draw()}} since it changes the current context.

The F\+L\+\_\+\+O\+P\+E\+N\+G\+L3 flag is required to access Open\+GL version 3 or more under the X11 and Mac\+OS platforms; it\textquotesingle{}s optional under Windows. See more details in opengl3.

{\footnotesize\ttfamily \mbox{\hyperlink{png_8h_aa8c59027f9ab2769342f248709d68d17}{void}} Fl\+\_\+\+Gl\+\_\+\+Window\+::show (\begin{DoxyParamCaption}{ }\end{DoxyParamCaption})\hspace{0.3cm}{\ttfamily [virtual]}}

Puts the window on the screen. Usually (on X) this has the side effect of opening the display.

If the window is already shown then it is restored and raised to the top. This is really convenient because your program can call \mbox{\hyperlink{class_fl___gl___window_a05f9c5395325f1f5a905207f0ccb994a}{show()}} at any time, even if the window is already up. It also means that \mbox{\hyperlink{class_fl___gl___window_a05f9c5395325f1f5a905207f0ccb994a}{show()}} serves the purpose of raise() in other toolkits.

\mbox{\hyperlink{class_fl___window_a0d75dfcdc8fb797f9247482da1e0152b}{Fl\+\_\+\+Window\+::show(int argc, char $\ast$$\ast$argv)}} is used for top-\/level windows and allows standard arguments to be parsed from the command-\/line.

\begin{DoxyNote}{Note}
For some obscure reasons \mbox{\hyperlink{class_fl___window_a908afc4420fd505e958f6ed394a86acc}{Fl\+\_\+\+Window\+::show()}} resets the current group by calling Fl\+\_\+\+Group\+::current(0). The comments in the code say \char`\"{}get rid of very common user bug\+: forgot end()\char`\"{}. Although this is true it may have unwanted side effects if you \mbox{\hyperlink{class_fl___gl___window_a05f9c5395325f1f5a905207f0ccb994a}{show()}} an unrelated window (maybe for an error message or warning) while building a window or any other group widget.
\end{DoxyNote}
